1. About this package

This package release note lists the changes, limitations and known issues of the contents of the package ve_engine_v2.0.0.

This package is part of the Nuance Vocalizer Expressive SDK for Windows. Please refer to the SDK release note for a description of the entire release set including deliverables, system requirements and installation instructions.

2. Changes

2.1. Enhancements

2.1.1. Change of voice models

  • VE 2.0.0 supports the new voice models vop1_sch1f22 and vop2_sch1f22. These are encoded with the OPUS audio codec, and they replace the Premium (full_155mrf22) and the Plus (dri80_1175mrf22) model in the regular offerings on VE 2.0.

  • The Standard (dri40_155mrf22) voice model is still supported on the engine, but it’s dropped as regular offering.

2.1.2. Voice operating points (vop)

  • The introduction of the parameter voice operating point affects several elements:

    • VE API: In the list of control parameters VE_PARAM_VOICE_OPERATING_POINT replaces VE_PARAM_VOICE_MODEL.

    • VE build: On the build command line the build option voice_operating_points replaces voice_models.

    • VE test drivers: On the command line the argument -O voice_operating_point replaces -m voice_model

  • A new control parameter voice operating point supersedes the parameter voice model, and it hides the combination of fe_cfg + voice_model. It defines an operating point from the user’s point of view: it delivers a target speech quality for a target cost of resources.

  • This allows the MNC operating points Embedded Compact (=cfg3+bet3) and MNC Embedded Pro (=cfg4+bet3) to coexist in the same VE stage.

  • The regular vops are named Embedded Compact (EC), Embedded Pro (EP), Embedded High (EH) and Premium High (PH). They are basically defined as:

2.1.3. Single data component per voice operating point

  • The VE stage has a single data component, a voice pack, that bundles all language and voice data to load and run a voice operating point on the engine. This covers the native language data and voice data, as well as optional foreign language data.

  • The voice pack carries a version suffix in its name. It is called like enu_ava_embedded-pro_1-0-0.dat; the underscore is used as field separator for lng, voice name, vop and version number (case 78872). This makes it easy to identify a voice pack in any deployment.

2.1.4. VE API changes

  • As mentioned above, the control parameter VE_PARAM_VOICE_OPERATING_POINT replaces VE_PARAM_VOICE_MODEL.

  • We introduced the type size_t in VE API functions and structs (instead of type LH_U32) for offset and size values (case 75916).

  • In function ve_ttsOpen() we dropped the deprecated argument UserData. The client is to use hOutDevInstance for user data instead.

2.1.5. Improved linguistic processing

  • For KOK the VE engine reuses the partofspeech-based parser (used in ARW for diacritizing the input) and segments words for KOK input (case 75913). It is driven by KOK v11 language data included in the KOK vops 1.0.0.

  • The MNC voices get multi-lingual skills. The engine allows an MNC voice using the ENU language data to transcribe English text fragments.

2.1.6. Template-based matching

  • The engine accepts a new type of tuning resource, called a template database, that it uses for template-based matching (case 73781).

  • This is a text-to-text transformation on the input to VE, suited e.g. for a navigation prompt, where the client has good control on the input generation. It gives full control on the use of APs and text in the context of the particular prompt, and scales well to large numbers of templates (order of 100 000).

  • You read the details in section "Prompt Templates" in the User’s Guide and Programmer’s Reference (ve_ugpr.pdf).

2.1.7. Tuning resources

  • We have simplified the order of execution of all rulesets: VE executes them most recently loaded to firstly loaded (case 67123).

  • The type of a ruleset is case-insensitive (case 55267).

  • A ruleset can take an output type (case 55267). This defines the <esc>\tn=$output_type\ to be set on the text fragment on completing the execution of the ruleset.

  • We have enabled the PCRE support for Unicode Character Properties (case 32718) on rulesets. As a result you can use \p{} in a regex to match a character on a property.

  • VE accepts loading an empty tuning resource (case 69342).

3. Limitations and known issues

3.1. Backwards compatibility

  • The Vocalizer Expressive v2.0.0 API is not backwards compatible with the API of earlier releases of Vocalizer for Automotive and Vocalizer Expressive.
